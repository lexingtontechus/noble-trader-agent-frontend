// Dynamic Regime Risk Management Platform

Regime-Aware
Risk Management API

A production-grade FastAPI service that detects market regimes with Hidden Markov Models and uses them to dynamically gate position sizes via the Kelly Criterion, compute real-time risk metrics (VaR/CVaR), and stream live regime snapshots and alerts over WebSocket and SSE.

⚡
Full Pipeline
One-shot: prices in, complete regime + sizing + risk JSON out.
⇄
WebSocket Stream
Push ticks and receive regime snapshots in real time, fully duplex.
→
SSE Alerts
Subscribe to regime-change alerts across all symbols via EventSource.
◈
Regime Matrix
9-cell vol × trend matrix with risk multipliers from 0.25× to 1.50×.

What it Does

The platform runs a three-stage pipeline on every price update:

01
Regime HMM
Dual Gaussian HMM
vol + trend states
→
02
Kelly Sizer
Fractional Kelly
vol-scaled + gated
→
03
Risk Engine
VaR / CVaR
stop + TP targets
→
04
Stream Out
WS / SSE / REST
+ regime alerts

Streaming Architecture

Each symbol maintains an independent StreamSession managed by the SessionRegistry singleton. Sessions have three concurrent responsibilities:

Responsibility Mechanism Latency
Tick path HMM predict → Kelly → VaR/CVaR → push to subscriber queues ~7 ms/tick
Background refit asyncio.run_in_executor — non-blocking EM fitting every N bars ~200–500 ms, async
Alert debounce 3-bar hysteresis before firing RegimeAlert 3 bars delay
✦
Multiple concurrent subscribers Any number of WebSocket clients and SSE consumers can subscribe to the same symbol session. The session pushes to an asyncio.Queue per subscriber. Slow consumers drop ticks (queue is bounded at 500 items) to prevent backpressure from blocking the tick path.

Core Concepts

Regime Detection

Two independent 3-state Gaussian HMMs are fitted on log-returns — one for volatility regimes, one for trend regimes. Each returns posterior state probabilities (not just Viterbi hard assignments), so confidence scores are smooth and differentiable.

Dimension States Features
Volatility low · med_low · med_high · high 20-bar & 5-bar realised vol + |return|
Trend strong_bear · bear · bull · strong_bull raw return + 10-bar & 30-bar cumulative return

States are semantically labelled by sorting the HMM component means — no manual label assignment needed. The combined regime_label (e.g. low_vol_bull) is produced by joining the dominant vol and trend states.

Kelly Criterion

Position sizes are computed using the continuous-returns Kelly formula: 

f* = μ / σ²

where μ is the mean daily log-return and σ² is the variance. The raw Kelly fraction then passes through three scaling stages:

Stage Field Formula
1. Fractional Kelly fractional_f full_kelly_f × kelly_fraction (default 0.5)
2. Vol scaling vol_scaled_f fractional_f × (target_vol / realised_annual_vol)
3. Regime gating regime_gated_f vol_scaled_f × risk_multiplier
⚠
Full Kelly can be extreme Full Kelly maximises long-run geometric growth but implies roughly 50% drawdowns. The 50% fractional Kelly default reduces drawdown risk while preserving most of the compounding benefit.

Risk Multiplier

Each regime maps to a risk_multiplier in the range [0.10, 1.75] using the formula: 

vol_scale   = {low: 1.75, med_low: 1.20, med_high: 0.70, high: 0.35}[vol_state]
trend_scale = {strong_bull: 1.00, bull: 0.88, bear: 0.60, strong_bear: 0.40}[trend_state]
risk_multiplier = clip(vol_scale × trend_scale, 0.10, 1.75)

VaR and CVaR

Risk metrics use historical (empirical) VaR/CVaR rather than parametric estimates. This is more accurate for fat-tailed return distributions common in financial assets.

Metric Definition
var_95 5th percentile of the return distribution (95% of days lose less than this)
cvar_95 Mean loss in the worst 5% of days (Expected Shortfall)
suggested_stop 2× CVaR95 — initial stop-loss suggestion
suggested_tp 3× CVaR95 — take-profit at 3:1 reward/risk

Alert Severity Levels

Severity Trigger condition Action suggested
critical New regime is high_vol_bear or high_vol_neutral Reduce exposure immediately
warning New regime contains bear or high (but not critical) Review position sizing
info All other transitions (e.g. neutral→bull) Log and monitor
✦
3-bar debounce Alerts only fire when the new regime persists for 3 consecutive bars. This prevents noise-driven micro-transitions from flooding downstream consumers with false alarms.

Regime Detection

POST /regime/detect Detect market regime from price series

Request Body

symbolstringoptional
Ticker symbol for labelling.
Default: "UNKNOWN"
pricesfloat[]required
Ordered close price series. Min 81 values, all positive.

Response Fields

vol_probsobject–
Posterior probabilities for low, medium, high volatility states. Sum to 1.0.
trend_probsobject–
Posterior probabilities for bear, neutral, bull trend states. Sum to 1.0.
regime_labelstring–
Combined label: {vol_state}_vol_{trend_state}. E.g. low_vol_bull.
confidencefloat–
max(vol_probs) × max(trend_probs). Range [0, 1]. High values indicate a clear, unambiguous regime.
risk_multiplierfloat–
Regime-derived scaling factor. Range [0.10, 1.75]. Used by the sizing engine.
n_bars_fittedint–
Number of price bars used in the HMM fit.

Kelly Sizing

POST /size/kelly Regime-aware Kelly position sizing

Request Body

pricesfloat[]required
Min 51 price bars. Used to compute log-returns and fit the HMM.
returnsfloat[]optional
Explicit return series. If provided, overrides returns computed from prices.
kelly_fractionfloatoptional
Fractional Kelly multiplier. 0.5 = Half-Kelly.
Default: 0.5 · Range: [0.1, 1.0]
target_volfloatoptional
Annualised target portfolio volatility for vol-scaling.
Default: 0.15
use_regimebooloptional
Whether to fit HMM and apply regime gating to the final fraction.
Default: true

Response Fields

full_kelly_ffloat–
Raw Kelly fraction f* = μ/σ², clipped to [0, 1].
fractional_ffloat–
After applying kelly_fraction multiplier.
vol_scaled_ffloat–
After vol-targeting: fractional_f × (target_vol / realised_annual_vol).
regime_gated_ffloat–
After regime multiplier. This equals recommended_f.
recommended_ffloat–
Final recommended position fraction in [0, 1]. Multiply by portfolio notional to get dollar size.
notesstring[]–
Automatic diagnostic notes (e.g. "Full Kelly > 50%", "Negative Sharpe").

Risk Analysis

POST /risk/analyse Full risk decomposition: VaR, CVaR, drawdown, stop/TP

Request Body

Extends SizeRequest with one additional field:

base_risk_limitfloatoptional
Max acceptable daily loss fraction at 1× leverage. Scaled by risk_multiplier to produce regime_max_loss.
Default: 0.02 · Range: (0, 0.5]

Response Fields

var_95 / var_99float–
1-day historical Value at Risk at 95% and 99% confidence. Expressed as a positive fraction.
cvar_95 / cvar_99float–
Expected Shortfall (mean loss in worst 5%/1% of days).
max_drawdownfloat–
Maximum peak-to-trough drawdown over the price series. Negative value.
sortino_ratiofloat–
Annual return / downside deviation. Penalises only negative volatility.
calmar_ratiofloat–
Annual return / |max drawdown|. Measures return per unit of worst-case loss.
regime_max_lossfloat–
Regime-adjusted daily loss limit: base_risk_limit × risk_multiplier.
suggested_stopfloat–
Suggested stop-loss: 2× CVaR95. Negative value (price drop).
suggested_tpfloat–
Suggested take-profit: 3× CVaR95. Positive value (3:1 reward/risk).
risk_budget_usedfloat–
recommended_f as a fraction of the max safe position. Values > 1.0 indicate over-allocation.

WebSocket Stream

WS /ws/{symbol} Full-duplex WebSocket stream

Client → Server Messages

All messages are JSON with a required type field.

seed
→ server
Seed the session. {"type": "seed", "symbol": "SPY", "prices": [...]}. Responds with {"type": "seeded", "ready": true, ...}.
tick
→ server
Push a price tick. {"type": "tick", "symbol": "SPY", "price": 512.34, "ts": 1710000512.0}. ts is optional.
subscribe
→ server
Switch to subscriber mode — receive all ticks produced by other producers on this symbol. {"type": "subscribe", "symbol": "SPY"}.
ping
→ server
Keepalive check. Responds with {"type": "pong", "ts": ...}.

Server → Client Messages

tick
← server
Full TickResponse JSON with "type": "tick" added.
alert
← server
Regime-change alert. {"type": "alert", "symbol", "previous", "current", "severity", "message"}.
seeded
← server
Seed confirmation: {"type": "seeded", "ready": true, "n_bars": 200, "refit_count": 1}.
buffering
← server
Warming up: {"type": "buffering", "n_bars": N}. Keep pushing ticks.
heartbeat
← server
Sent every 30 seconds of inactivity in subscriber mode to keep the connection alive.
error
← server
{"type": "error", "detail": "..."}. Connection remains open.

Example — Python

python — asyncio + websockets 
import asyncio, json, websockets

async def main():
    url = "ws://localhost:8000/ws/SPY"
    async with websockets.connect(url) as ws:

        # Seed
        await ws.send(json.dumps({
            "type": "seed", "symbol": "SPY",
            "prices": historical_prices
        }))
        ack = json.loads(await ws.recv())
        assert ack["ready"] is True

        # Stream ticks
        async for price in live_price_feed():
            await ws.send(json.dumps({
                "type": "tick", "symbol": "SPY", "price": price
            }))
            msg = json.loads(await ws.recv())
            if msg.get("regime_changed"):
                print(f"REGIME CHANGE → {msg['regime_label']}")
            print(f"f*={msg['recommended_f']:.4f}  VaR95={msg['var_95']:.4f}")

asyncio.run(main())

SSE Streams

Server-Sent Events are ideal for browser dashboards and read-only consumers. The connection is one-way (server → client) and reconnects automatically on drop.

SSE /sse/{symbol} Per-symbol regime snapshot stream

Pushes a full TickResponse JSON on every tick processed for symbol. Requires the session to be seeded first. Sends a comment heartbeat (: heartbeat) every 15 seconds of inactivity.

javascript — browser EventSource 
const es = new EventSource('http://localhost:8000/sse/SPY');

es.onopen  = () => console.log('connected');
es.onerror = () => console.log('reconnecting...');

es.onmessage = event => {
  const tick = JSON.parse(event.data);
  if (tick.type === 'connected') return;

  document.getElementById('regime').textContent  = tick.regime_label;
  document.getElementById('size').textContent    = (tick.recommended_f * 100).toFixed(1) + '%';
  document.getElementById('var95').textContent   = (tick.var_95 * 100).toFixed(2) + '%';
};
SSE /sse/alerts Global regime-change alert broadcast

Receives RegimeAlert events across all active symbol sessions. Only fires on regime transitions that persist for 3+ bars. Use for portfolio-level risk dashboards or notification systems.

alert event payload
{
  "type":     "alert",
  "symbol":   "SPY",
  "ts":       1710000512.0,
  "previous": "low_vol_bull",
  "current":  "high_vol_bear",
  "severity": "critical",
  "message":  "⚡ REGIME CHANGE [SPY]: low_vol_bull → high_vol_bear | risk_mult=0.25× | recommended_f=0.0812"
}
⚠
critical severity triggers when the new regime is high_vol_bear or high_vol_neutral — the two highest-risk states in the matrix. Consider subscribing to this stream in any automated risk management system.

Why not pynamical or simupy?

pynamical is a chaos theory / bifurcation visualisation library (Lyapunov exponents, logistic maps). simupy is a block-diagram ODE solver for dynamical systems. Both are simulation tools, not statistical inference engines.

For a trading risk service where the goal is real-time probabilistic regime classification from market data, hmmlearn + scipy provide the right combination of statistical rigour and production performance. Either library could be added as an extension endpoint — simupy for ODE-based regime transition simulation, pynamical for portfolio chaos analysis.

Observation Builder

// core/obs_builder.py — InferenceObservationBuilder

Constructs a 24-feature observation vector for every price bar. Features 14–19 are the Markov features — live HMM posterior probabilities that must be computed from the fitted RegimeHMM, never from a uniform prior.

⚠
Uniform-prior bug If Markov features (indices 14–19) show constant values (~0.33), the HMM state is not being passed into the observation. Always use InferenceObservationBuilder (stateful class), never the standalone build_observation() function. Call obs.is_markov_uniform() to detect this at runtime.

Two builder paths

Path Class / function Markov features Use when
✓ Stateful InferenceObservationBuilder.build() Dynamic from fitted HMM All production use
✗ Standalone build_observation() (do not use) Uniform [0.25, 0.25, 0.25, 0.25] Never — bug risk

Quick Start

python

Markov Guard

Features 14–19 are the most diagnostically important in the observation vector. When they are uniform, the HMM is not contributing any signal — the policy is flying blind on regime information.

The Uniform-Prior Bug

⚠
Symptom: Markov features (indices 14–19) show constant horizontal bands at ~0.33 in observation heatmaps, or is_markov_uniform() returns True.

Cause: The HMM state is not being passed into the observation builder — it defaults to [0.25, 0.25, 0.25, 0.25] (uniform prior).

Fix: Use InferenceObservationBuilder (stateful class) and call seed() before any build() call.

Failed Patterns (do not use)

Anti-pattern Why it fails
Calling build_observation(prices) standalone No HMM state passed → uniform [0.25]*4
Creating a new builder per bar HMM re-fitted cold each time, no state continuity
Calling build() without seed() HMM not yet fitted → all-zero sentinel (not uniform, but still useless)
Disabling Markov features Breaks model compatibility and removes regime signal entirely

Detection at Runtime

python — guard check
obs = builder.build(prices, high, low)

# Method 1: built-in guard
if obs.is_markov_uniform():
    raise RuntimeError("Uniform-prior bug detected — check seed() was called")

# Method 2: manual std check
markov_std = obs.markov_features.std()
assert markov_std > 0.05, f"Markov features uniform: std={markov_std:.4f}"

# Method 3: visualise over time (heatmap)
# If indices 14-19 show flat horizontal bands → bug is active

Degenerate HMM Recovery

The RegimeHMM._sanitize_model() method (called automatically after every EM fit) handles three degenerate failure modes:

Failure Cause Fix applied
NaN startprob_ All observations assigned to one state; others get 0 probability mass Replace with uniform 1/n
Zero transmat_ rows State never transitioned to from another state during EM Replace row with uniform 1/n
NaN/zero _covars_ Degenerate Gaussian (all identical observations in state) Replace bad entries with 1e-3

After sanitization, predict_proba() always returns finite values. If posteriors are still NaN after sanitization, predict() falls back to uniform and logs a warning — it never propagates NaN into the observation vector.

Redis Persistence

// services/redis_persistence.py — REDIS_URL env var

Optional Redis Streams persistence layer. When REDIS_URL is set, price buffers survive process restarts and can be shared across multiple uvicorn workers. Without it, the system operates entirely in-memory with no performance overhead.

✦
Zero configuration overhead. If REDIS_URL is not set, all RedisPersistence methods are no-ops. No code changes needed — just add the env var to enable.

How it works

Event Redis operation Key pattern
App startup XREVRANGE last N entries → hydrate buffer regime:prices:{symbol}
Each tick XADD with approximate MAXLEN cap regime:prices:{symbol}
Session seed HSET session config regime:meta:{symbol}
Session remove DEL meta (price stream preserved by default) regime:meta:{symbol}

Startup restore flow

main_v3.py lifespan
# On startup, main_v3.py automatically:
1. Lists all symbols with persisted streams in Redis
2. Loads session config from Redis meta hash
3. Restores price buffer via XREVRANGE
4. Re-fits HMM if ≥ 81 bars are available
5. Sessions are immediately ready for incoming ticks

Error policy

All Redis operations are wrapped in try/except. A Redis failure never raises in the tick path — it logs a warning at DEBUG level and the system continues with in-memory state. Redis is persistence, not availability.

Multi-worker deployment

⚠
With Redis enabled, multiple uvicorn workers can share price history — each worker restores the same buffer on startup. However, the in-memory SessionRegistry (and therefore WebSocket subscribers) is still per-worker. For true multi-worker real-time streaming, use a single worker or add a pub/sub layer (e.g. Redis Pub/Sub for tick fan-out).

Dynamic Masaniello Sizer

// core/position_sizer.py — DynamicMasanielloSizer

Stateful per-trade position sizing combining Masaniello batch progress, model quality, drawdown protection, and volatility scaling into a single formula. Fully integrated with RegimeSnapshot.

The Formula

f_i = β × (0.5 + M_i) × Q_i × DD_i × V_i
Factor Symbol Formula Clamp Purpose
Base risk β SizingConfig.base_risk (default 0.50%) — Risk fraction anchor
Masaniello M_i (W − w) / (N − i + 1) [0.0, 1.5] Urgency: wins needed ÷ trades left
Quality Q_i prob_factor × regime_factor × conf_factor product of clamps Combined edge × regime × confidence gate
Drawdown DD_i clip(1 − dd / max_dd, 0.25, 1.0) [0.25, 1.0] Reduce size; never zero (floor 0.25)
Volatility V_i clip(ATR_base / ATR_current, 0.5, 1.5) [0.5, 1.5] Inverted ATR — high vol shrinks size
ℹ
Why (0.5 + M_i)? This term guarantees a minimum sizing multiplier of 0.5× base risk even when M_i = 0 (target already met, or first trade in a relaxed batch). Position never collapses to zero from the Masaniello factor alone.

Q_i — Quality Factor detail

Sub-factor Formula Clamp Source
prob_factor clip((p_win − min_prob) / 0.10, 0.0, 1.5) [0.0, 1.5] Estimated win probability
regime_factor clip(regime_quality, 0.0, 1.5) [0.0, 1.5] RegimeSnapshot.risk_multiplier
conf_factor clip(state_confidence, 0.0, 1.0) [0.0, 1.0] RegimeSnapshot.confidence

Maximum Q_i = 1.5 × 1.5 × 1.0 = 2.25 (all factors maxed). Q_i = 0 when any sub-factor is 0 — the formula acts as a quality gate before any risk is allocated.

Pre-flight Gates

Checked in priority order before the formula runs. Each gate fires with a specific .reason string:

# Gate Condition
1 Equity equity > 0
2 Stop distance stop_distance_price > 0
3 Point value point_value > 0
4 Regime floor regime_quality ≥ regime_floor (default 0.50)
5 Min probability p_win ≥ min_prob (default 0.50)
6 Min R/R reward_risk ≥ min_rr (default 2.50)
7 Batch halt batch not already halted
8 Drawdown halt batch.drawdown < batch_halt_dd (default 5%)
9 Edge E = p·R − (1−p) > 0

Usage

python — with RegimeSnapshot
from regime_platform.core.position_sizer import (
    DynamicMasanielloSizer, SizingConfig, BatchState
)

cfg   = SizingConfig(base_risk=0.005, max_drawdown=0.10, batch_size=5, target_wins=3)
sizer = DynamicMasanielloSizer(cfg)
sizer.batch.peak_equity = equity

# Size one trade from a live RegimeSnapshot
result = sizer.size_from_snapshot(
    snap=snap,                     # from RegimeHMM.predict()
    equity=100_000,
    stop_distance_price=2.50,    # stop in price units
    point_value=50.0,            # $ per point
    p_win=0.62,
    reward_risk=2.5,
    atr_baseline=12.0,
    atr_current=10.0,
)
print(result.summary())
# [✓ ALLOWED]  f=0.550%  $550  units=5.50  contracts=5  ...

# After trade resolves, record outcome
sizer.batch.record(result, outcome=True, pnl=625.0)

# After N trades, reset for next batch
sizer.batch.reset(equity=100_625)

curl_test.sh

// bash — 6 assertions · 200-bar SPY payload · edge cases

Self-contained bash script. Seeds a 200-bar multi-regime SPY payload, calls POST /analyse/full, asserts all 12 response fields, then validates two error cases (too few bars → 422, negative price → 422).

bash — run
chmod +x curl_test.sh
./curl_test.sh                         # localhost:8000
./curl_test.sh http://my-host:8000     # custom URL

What it checks

# Check Expected
1 GET /health 200 OK
2 POST /analyse/full with 200 bars 200 OK
3 12 response fields present All non-null
4 POST /analyse/full with 6 bars 422 Unprocessable
5 POST /analyse/full with negative price 422 Unprocessable
✦
Install jq for coloured JSON output. Without it, the script falls back to python3 -m json.tool automatically.

stream_spy_ticks.py

// Python — seed + live ticks · colour output · regime detection

Seeds SPY with 150 historical bars then pushes 50 live ticks one-by-one via POST /stream/tick, printing a formatted TickResponse for each. Pure stdlib + numpy — no extra dependencies.

bash — variants
# Default: 150 seed bars, 50 ticks, mixed regime
python stream_spy_ticks.py

# Crisis scenario, slower tick rate
python stream_spy_ticks.py --regime crisis --ticks 80 --interval 0.3

# Benchmark: 200 ticks, no delay
python stream_spy_ticks.py --ticks 200 --interval 0 --seed-bars 300

# Log to file (no colour)
python stream_spy_ticks.py --no-color > tick_log.txt

# Filter regime changes only (pipe raw JSON to jq)
python stream_spy_ticks.py --json | jq '. | select(.regime_changed == true)'

sse_alert_monitor.html

// Browser — EventSource · real-time · console logging

Open in any browser. Connects to GET /sse/alerts and displays each regime-change alert with severity colour-coding, probability bars, and a live stats counter. Every message is also logged to the browser console.

bash — open
open sse_alert_monitor.html          # macOS
xdg-open sse_alert_monitor.html      # Linux

Console output by severity

Severity Console method Trigger
critical console.error high_vol_bear or high_vol_neutral — badge pulses 3×
warning console.warn Any bear or high-vol regime entry
info console.info Benign transitions (e.g. neutral → bull)
ℹ
Console tip: Open DevTools → Console before connecting. Every raw JSON message is logged with console.log('[SSE]', msg) so you always have the full payload for inspection. Critical alerts appear in red, warnings in yellow — matching the browser's native console severity colouring.

Features

  • ✓ EventSource auto-reconnect with error distinction
  • ✓ Live stats: info / warning / critical counts, heartbeats, uptime
  • ✓ Regime transition arrow with colour-coded labels
  • ✓ risk_mult and recommended_f extracted from alert message
  • ✓ Log capped at 300 entries — no memory growth
  • ✓ All server strings sanitised via escHtml() (XSS guard)
  • ✓ Enter key connects / disconnects
  • ✓ Configurable URL field — works with any CORS-enabled SSE endpoint
